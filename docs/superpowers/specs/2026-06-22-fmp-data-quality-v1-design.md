# FMP Data Quality v1 Design

## Decision: Is This Worth Doing?

Yes. This is a necessary foundation before adding semiconductor fundamentals, TradingView integration, or options/sentiment analytics. The system already has signals, LOTS sizing, risk exits, factor research, Telegram alerts, and AI/Supervisor review. Those layers are only useful when the underlying market data is reliable.

This is especially important because real positions can include broker symbols that are not in the configured universe or are not covered by the current provider. If the system cannot price an open position, it must treat that as a data repair problem, not as a normal "no signal" condition.

## Goals

- Build a product-grade data quality layer around current provider checks.
- Check configured universe, leveraged/tactical ETFs, benchmarks, and open positions.
- Preserve broker tickers while using symbol aliases for provider lookups.
- Classify each ticker into an action status:
  - `tradable`
  - `research_only`
  - `data_fix_required`
  - `blocked`
- Produce a CLI summary and optional HTML report.
- Make data quality usable by future reports, monitor, and Supervisor review.
- Keep production buy/sell logic unchanged in this phase.

## Non-Goals

- No options chain, option wall, put/call ratio, IV rank, skew, or gamma exposure.
- No TradingView webhook or Pine Script integration.
- No fundamental factor integration.
- No automatic ticker correction.
- No automatic trading or broker API.
- No change to signal scoring thresholds.

## Why This Comes Before Other Features

The next layers all depend on data reliability:

- Semiconductor fundamental factors require stable provider coverage.
- TradingView alerts are only useful if they can be reconciled with the system's tickers.
- Options/sentiment analytics will be noisy unless the underlying equity data is already trustworthy.
- AI/Supervisor review should not approve ideas when the provider coverage is blocked.

## Scope

### Inputs

- `config.universe.tickers`
- `config.universe.leveraged_tickers`
- `config.universe.tactical_tickers`
- `config.universe.benchmarks`
- open positions from SQLite
- symbol aliases from SQLite
- provider checks from existing `check_provider_data`

### Ticker Roles

Each ticker can have one or more roles:

- `universe`
- `open_position`
- `benchmark`
- `leveraged_etf`
- `tactical`

Open positions have highest priority. If an open position has no usable price, the output must be `data_fix_required` or `blocked`, not a normal observation.

### Asset Classes

Classify tickers into:

- `stock`
- `etf`
- `leveraged_etf`
- `benchmark`
- `unknown`

The classification can be rule based in v1:

- configured leveraged/tactical tickers -> `leveraged_etf`
- configured benchmarks -> `benchmark`
- known ETF-style tickers such as `QQQ`, `SMH`, `SPY`, `SOXX`, `TQQQ`, `SOXL`, `USD` -> ETF classes
- otherwise -> `stock`

### Provider Status

Use current provider checks as the first implementation layer:

- `provider`
- `rows`
- `first_date`
- `last_date`
- `latest_close`
- `stale_days`
- `ok`
- provider message

Classification rules:

- `tradable`: enough recent daily data and no blocking issue.
- `research_only`: enough data for context, but not enough for live trading confidence.
- `data_fix_required`: open position or configured ticker has no usable data and may need alias/provider repair.
- `blocked`: provider failure, missing required API key, stale data beyond threshold, or insufficient history for strategy use.

## Public Equity Risk Fields

Every data quality record should carry a compact risk explanation:

- `intended_alpha`: why this ticker is in the system, such as AI/semi trend, benchmark, tactical ETF, or held-position monitoring.
- `unwanted_risk`: what bad decision could happen if this data is trusted incorrectly.
- `retained_exposure`: what exposure remains relevant if this is an open position.
- `binding_constraint`: the actual data issue blocking or limiting use.
- `liquidity_exit_posture`: whether the system can still evaluate hold/trim/exit, or must require manual repair first.

These are not generated by the Codex plugin at runtime. They are deterministic text fields in the Python system using Public Equity risk discipline.

Core functions:

```text
collect_data_quality_universe(config)

write_data_quality_report(report, out_path)
```

## CLI

Add:

```bash
bin/quant-ai-local data-quality --config config/default.yaml
```

Optional HTML:

```bash
bin/quant-ai-local data-quality --config config/default.yaml --out outputs/data_quality_report.html
```

Exit code:

- `0` if no `blocked` records.
- `1` if any record is `blocked`.
- `0` when only `research_only` records exist.
- `1` when open positions are `data_fix_required`.

## Report Requirements

CLI output:

- total tickers checked;
- counts by status;
- open-position blockers first;
- each blocked/data-fix ticker with role, provider symbol, rows, last date, and binding constraint.

HTML output:

- top summary cards;
- open-position data blockers;
- table of all records;
- alias repair guidance;
- clear warning that blocked tickers cannot support buy/add signals.

## Error Handling

- Missing `FMP_API_KEY` must not crash.
- Provider failure must become a `blocked` or `data_fix_required` record.
- Alias lookup failures must fall back to broker symbol and mark alias status.
- Empty provider responses must be explicit.
- HTML report should still render even when all provider calls fail.

## Testing Plan

Use TDD before implementation.

Tests should cover:

- open positions are included even if not in `config.universe`.
- broker symbol alias uses data symbol for provider checks while preserving broker symbol in output.
- missing open-position data becomes `data_fix_required`.
- missing non-held universe ticker becomes `blocked` or `research_only` depending on configured rule.
- benchmarks and leveraged ETFs are classified separately.
- missing `FMP_API_KEY` creates explicit provider failure records.
- CLI returns non-zero when open positions are blocked.
- HTML report includes data blockers, alias guidance, and Public Equity risk fields.

## Acceptance Criteria

```bash
.venv/bin/python -m pytest tests/test_data_quality.py -q
bin/quant-ai-local data-quality --config config/default.yaml --out outputs/data_quality_report.html
.venv/bin/python -m pytest -q
```

The report must answer:

- Which tickers are tradable?
- Which tickers are research-only?
- Which tickers need alias or provider repair?
- Which tickers block live signals?
- Are any open positions missing usable data?
- Are tactical/leveraged ETFs covered?
- What is the binding data constraint for each blocked ticker?

## Follow-Up Gate

Before building semiconductor fundamental factors, TradingView integration, or options/sentiment analytics, ask:

1. Does the data quality report show stable coverage for the relevant ticker set?
2. Are real open positions covered or explicitly mapped by alias?
3. Can the next feature use this data without silently approving bad signals?
4. Will the next feature change a real sizing, add, trim, or exit decision?

If the answer is no, fix data quality first.
